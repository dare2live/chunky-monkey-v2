## 1. 项目目标

把系统正式收口为一条“决策驱动”的完整研究主线：

`更新数据 -> 生成研究池 -> 行业/个股解释 -> Qlib 排序增强 -> Setup 快照 -> 历史验证 -> 持续迭代`

系统最终目标不是只多几个分数，而是同时做到：
- 指标口径统一
- 数据链路可回放、可重算、可审计
- 页面组织贴合真实使用流程
- 旧兼容逻辑不再主导排序和解释
- 股东挖掘主链与 ETF 主链都按同一套决策链工作

## 2. 系统定位

系统正式定义为：

`以十大流通股东挖掘为入口、以机构行业能力为发现器、以公司质量为确认器、以阶段判断为过滤器、以 Qlib 为排序器的 A 股研究系统`

四层主分固定为：
- `Discovery`
- `Quality`
- `Stage`
- `Forecast`
- `Composite`

系统当前有两条正式业务主线：
- 股东挖掘主链：`东财股东事实 -> Discovery -> Quality -> Stage -> Qlib 排序 -> Setup 快照 -> 验证`
- ETF 主链：`mootdx ETF/指数底座 -> 市场温度 -> 低频宏观情景约束 -> Qlib ETF 排序 -> ETF 页面 -> ETF 验证`

对应页面角色固定为：
- 工作台：入口页，只负责更新状态、候选入口、快照入口
- 股票研究页：主研究页，承担排序、筛选、解释
- 行业页：研究背景页，承担行业结构和行业后验
- 个股详情页：解释页，承担四卡拆解
- Qlib 页：工具页，只负责训练、参数、模型状态
- 验证页：复盘页，承担快照后验、新旧排序对照、异常样本

## 4. 推进原则

### 4.1 按决策链与使用流程推进

后续工作不再按“模块想到哪里做到哪里”推进，而是按“决策链 -> 使用流程”推进：

1. 每日更新是否稳定
2. 发现决策是否可信
3. 质量决策是否稳定
4. 阶段决策是否可解释
5. Qlib 排序与模型验证是否真的接管预测问题
6. 研究页和验证页是否顺着决策链组织
7. legacy 是否可以安全退役

### 4.2 单点计算

所有关键变量坚持“单点计算，多处复用”：
- 同一个业务事实只在一个后端模块里计算
- 计算结果写回统一事实层或集市层
- 前端只展示，不重复推导主变量

### 4.3 先保真，再提效，再清理

执行顺序固定为：
- 先补数据和评分口径
- 再优化页面和验证链路
- 最后退役兼容层

### 4.4 数据源优先级

后续数据增强默认遵守：
- `mootdx` 优先
- `AKShare` 补充
- 预测与预测验证默认 `Qlib` 优先

原因：
- `mootdx` 更稳定、更快，适合做主底座和连续口径
- `AKShare` 适合补事件、扩展字段和相对比较层
- 对稳定性和更新链要求高的信号，优先从 `mootdx` 已有数据派生，不优先依赖脆弱网页接口
- 对未来收益排序、模型验证、因子贡献这类问题，优先直接复用完整版 `Qlib` 的真实模型产物，不再并行造一套自定义预测系统

### 4.5 架构评审采纳方式

结合 `docs/DISCUSSION_ARCHITECTURE_REVIEW.md` 与项目当前真实状态，后续按 3 类处理：

立即采纳：
- 用“决策优先定目标，数据优先定当前实现深度，并把置信度向上传播”作为正式原则
- `mootdx` 优先承担主底座
- 未来收益相关问题优先交给 `Qlib workflow / record / backtest`
- ETF 主链先做“市场温度 + 宏观约束 + 排序”三层，不把宏观叙事直接当交易结论

分阶段采纳：
- `fetch_index_kline()` 接入真实指数底座
- `SignalRecord / SigAnaRecord / PortAnaRecord` 接入自动训练主流程
- 置信度传播、动态权重、FeatureCoverage 元数据
- `scoring.py` 继续拆分成更清晰的 scorer 模块

暂不优先：
- 为了目录整齐而做的大规模 client 重命名
- 不带业务收益的纯重构搬家
- 在 Qlib 已能承担的场景里继续自造平行预测系统
